When you\textquotesingle{}re ready to make this README your own, just edit this file and use the handy template below (or feel free to structure it however you want -\/ this is just a starting point!). Thank you to \href{https://www.makeareadme.com/}{\texttt{ makeareadme.\+com}} for this template.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md11}{}\doxysubsection{\texorpdfstring{Suggestions for a good README}{Suggestions for a good README}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md11}
Every project is different, so consider which of these sections apply to yours. The sections used in the template are suggestions for most open source projects. Also keep in mind that while a README can be too long and detailed, too long is better than too short. If you think your README is too long, consider utilizing another form of documentation rather than cutting out information.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md12}{}\doxysubsection{\texorpdfstring{Name}{Name}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md12}
Choose a self-\/explaining name for your project.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md13}{}\doxysubsection{\texorpdfstring{Description}{Description}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md13}
Let people know what your project can do specifically. Provide context and add a link to any reference visitors might be unfamiliar with. A list of Features or a Background subsection can also be added here. If there are alternatives to your project, this is a good place to list differentiating factors.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md14}{}\doxysubsection{\texorpdfstring{Badges}{Badges}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md14}
On some READMEs, you may see small images that convey metadata, such as whether or not all the tests are passing for the project. You can use Shields to add some to your README. Many services also have instructions for adding a badge.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md15}{}\doxysubsection{\texorpdfstring{Visuals}{Visuals}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md15}
Depending on what you are making, it can be a good idea to include screenshots or even a video (you\textquotesingle{}ll frequently see GIFs rather than actual videos). Tools like ttygif can help, but check out Asciinema for a more sophisticated method.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md16}{}\doxysubsection{\texorpdfstring{Installation}{Installation}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md16}
Within a particular ecosystem, there may be a common way of installing things, such as using Yarn, Nu\+Get, or Homebrew. However, consider the possibility that whoever is reading your README is a novice and would like more guidance. Listing specific steps helps remove ambiguity and gets people to using your project as quickly as possible. If it only runs in a specific context like a particular programming language version or operating system or has dependencies that have to be installed manually, also add a Requirements subsection.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md17}{}\doxysubsection{\texorpdfstring{Usage}{Usage}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md17}
Use examples liberally, and show the expected output if you can. It\textquotesingle{}s helpful to have inline the smallest example of usage that you can demonstrate, while providing links to more sophisticated examples if they are too long to reasonably include in the README.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md18}{}\doxysubsection{\texorpdfstring{Support}{Support}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md18}
Tell people where they can go to for help. It can be any combination of an issue tracker, a chat room, an email address, etc.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md19}{}\doxysubsection{\texorpdfstring{Roadmap}{Roadmap}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md19}
If you have ideas for releases in the future, it is a good idea to list them in the README.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md20}{}\doxysubsection{\texorpdfstring{Contributing}{Contributing}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md20}
State if you are open to contributions and what your requirements are for accepting them.

For people who want to make changes to your project, it\textquotesingle{}s helpful to have some documentation on how to get started. Perhaps there is a script that they should run or some environment variables that they need to set. Make these steps explicit. These instructions could also be useful to your future self.

You can also document commands to lint the code or run tests. These steps help to ensure high code quality and reduce the likelihood that the changes inadvertently break something. Having instructions for running tests is especially helpful if it requires external setup, such as starting a Selenium server for testing in a browser.\hypertarget{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md21}{}\doxysubsection{\texorpdfstring{Authors and acknowledgment}{Authors and acknowledgment}}\label{md__2_users_2cailleyfactor_2_documents_2cf593_2_r_e_a_d_m_e_autotoc_md21}
